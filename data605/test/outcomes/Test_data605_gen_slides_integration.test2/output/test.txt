#### Version Control Systems (1/2)

- A **\textcolor{red}{Version Control System}** (VCS) is a system that allows to:
  - Record changes to files
  - Recall specific versions later (like a "file time-machine")
  - Compare changes to files over time
  - Track *who* changed *what* and *when* and *why*

- **\textcolor{darkgreen}{Simplest "VCS"}**
  - Make a copy of a dir and add
    - \textcolor{blue}{\texttt{\_v1}} (bad); or
    - a timestamp \textcolor{blue}{\texttt{\_20220101}} (better)
  - **\textcolor{blue}{Cons}**: It kind of works for one person, but doesn't scale

#### Version Control Systems (2/2)

::: columns
:::: {.column width=55%}
- **\textcolor{red}{Centralized VCS}**
  - E.g., \textcolor{blue}{\texttt{Perforce}}, \textcolor{blue}{\texttt{Subversion}}
  - A server stores the code, clients connect to it
  - **\textcolor{darkgreen}{Cons}**: If the server is down, nobody can work

#### Git
- **\textcolor{red}{Almost everything is local for Git}**
  - History stored locally in each node
  - Diff-ing files done locally
    - Centralized VCS requires server access
  - Commit to local copy
    - Upload changes with network connection

- **\textcolor{darkgreen}{Almost everything is undoable in Git}**
  - No data corruption
    - Everything checksummed
  - Nothing lost
  - Disclaimer:
    - Commit (at least locally) or stash
    - Know how to do it to avoid "git hell"

- **\textcolor{blue}{Git is a mini key-value store with a VCS built on top}**
  - Exactly true
  - Two layers:
    - "porcelain": key-value store for file-system
    - "plumbing": VCS layer

#### Sections of a Git Project
::: columns
:::: {.column width=55%}

- There are 3 main sections of a Git project
  - **\textcolor{red}{Working tree}** (aka checkout)
    - Version of code on the filesystem for use and modification
  - **\textcolor{darkgreen}{Staging area}** (aka cache, index)
    - File in \textcolor{blue}{\texttt{.git}} storing info for the next commit
  - **\textcolor{blue}{Git directory}** (aka .git)
    - Stores metadata and objects (like a DB)
    - The repo itself with all history
    - Cloning gets you the project's \textcolor{blue}{\texttt{.git}}

#### States of a File in Git

::: columns
:::: {.column width=55%}
- Each file can be in 4 states from Git point-of-view
  - **\textcolor{red}{Untracked}**: files not under Git version control
  - **\textcolor{darkgreen}{Modified}**: changed files, not committed yet
  - **\textcolor{blue}{Staged}**: marked modified files for next commit
  - **\textcolor{violet}{Committed}**: data stored in local DB

#### Git Tutorial
- **\textcolor{red}{Git tutorial on class repo}**
  - Follow the \textcolor{blue}{\texttt{README}}

- **\textcolor{orange}{How to use a tutorial}**
  - Type commands one-by-one
    - Avoid copy-paste
  - Observe results
    - Understand each line
  - Experiment
    - _"What happens if I do this?"_
    - _"Does the result match my mental model?"_
  - Learn command line before GUI
    - GUIs hide details and you become dependent on it

- Go through **\textcolor{darkgreen}{recommended Git book}** and try all examples
  - Use online tutorials

- Build your own **\textcolor{teal}{cheat sheet}**
  - Reuse others' cheat sheets only if familiar

- Achieve **\textcolor{cyan}{mastery of basic tools}**
  - Bash, Git, editor
  - Python, Pandas
  - ...


#### Git Daily Use
- Check out a project (\textcolor{blue}{\texttt{git clone}}) or start from scratch (\textcolor{blue}{\texttt{git init}})
  - Only once per Git project client

- **\textcolor{red}{Daily routine}**
  - Modify files in working tree (\textcolor{blue}{\texttt{vi ...}})
  - Add files (\textcolor{blue}{\texttt{git add ...}})
  - Stage changes for next commit (\textcolor{blue}{\texttt{git add -u ...}})
  - Commit changes to \textcolor{blue}{\texttt{.git}} (\textcolor{blue}{\texttt{git commit}})

- **\textcolor{blue}{Use a branch to group commits together}**
  - Isolate code from changes in \textcolor{blue}{\texttt{master}}
  - Merge \textcolor{blue}{\texttt{master}} into branch
  - Isolate \textcolor{blue}{\texttt{master}} from changes
  - Pull Request (PR) for code review
  - Merge PR into upstream

#### Git Remote
- **\textcolor{red}{Remote repos}**: versions of the project hosted online
  - Manage remote repos to collaborate
  - Push/pull changes
  - \textcolor{blue}{\texttt{git remote -v}}: show remotes
  - \textcolor{blue}{\texttt{git fetch}}: pull data from remote repo you don't have locally
  - \textcolor{blue}{\texttt{git pull}}: shorthand for \textcolor{blue}{\texttt{git fetch origin}} + \textcolor{blue}{\texttt{git merge master --rebase}}
  - \textcolor{blue}{\texttt{git push <REMOTE> <BRANCH>}}: push local data to remote
    - E.g., \textcolor{blue}{\texttt{git push origin master}}

- Multiple forks of the same repo with different policies
  - E.g., read-only, read-write

- If someone pushed to remote, you **\textcolor{blue}{can't push changes immediately}**:
  - Fetch changes
  - Merge changes to your branch
  - Resolve conflicts, if needed
  - Test project sanity (e.g., run unit tests)
  - Push changes to remote

#### Git Tagging
- Git allows marking points in history with a **\textcolor{red}{tag}**
  - E.g., release points
  - Check out a tag

- Enter detached \textcolor{blue}{\texttt{HEAD}} state
  - Committing won't add changes to the tag or branch
  - Commit will be "unreachable," reachable only by commit hash

#### Git Internals
::: columns
:::: {.column width=55%}

- **\textcolor{red}{Understand Git only if you understand its data model}**
  - Git is a key-value store with a VCS interface
  - Key = hash of a file
  - Value = content of a file

- **\textcolor{blue}{Git objects}**
  - Commits: pointers to the tree and commit metadata
  - Trees: directories and mapping between files and blobs
  - Blobs: content of files

- Refs:
  - Easy: 
[\textcolor{blue}{\underline{Understanding Git Data Model}}](https://konrad126.medium.com/https-medium-com-zspajich-understanding-git-data-model-95eb16cc99f5)
  - Hard-core: 
[\textcolor{blue}{\underline{Git internals}}](https://git-scm.com/book/en/v2/Git-Internals-Git-Objects)
